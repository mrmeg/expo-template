# Spec: Ignore Generated Coverage Output

**Status:** Ready
**Priority:** Medium
**Scope:** Client

---

## What
Prevent the Jest coverage output generated by `bun run test:ci` from dirtying the working tree. The task should add the missing ignore rule and verify that the existing test command remains unchanged.

## Why
The template's recommended CI test command should be safe to run repeatedly from a clean checkout. Generated coverage files are useful locally but should not appear as untracked project changes.

## Current State
`package.json` defines `test:ci` as `jest --ci --coverage --forceExit`. Running it creates a top-level `coverage/` directory. `.gitignore` ignores `dist/`, `.expo/`, native folders, and env files, but does not ignore `coverage/`.

## Changes
1. Add coverage output to `.gitignore`.
   - Ignore `coverage/` at the repository root.
   - Keep existing ignore sections organized.

2. Verify generated output behavior.
   - Run `bun run test:ci`.
   - Confirm `git status --short` does not show `coverage/`.
   - Remove any locally generated coverage output after verification if needed.

3. Update docs if they mention coverage artifacts.
   - Align `CONTRIBUTING.md` testing notes if necessary.

## Acceptance Criteria
1. `coverage/` is ignored by git.
2. `bun run test:ci` still passes.
3. Running `bun run test:ci` from a clean worktree does not leave untracked coverage files.
4. No generated coverage files are committed.

## Constraints
- Do not remove coverage generation from `test:ci`.
- Do not ignore source or test files broadly.
- Do not commit local coverage reports.

## Out of Scope
- Enforcing coverage thresholds.
- Changing Jest reporters.
- Adding hosted coverage upload.

## Files Likely Affected
Client / tooling:
- `.gitignore`

Docs:
- `CONTRIBUTING.md` only if testing instructions need clarification

## Edge Cases
- Existing local `coverage/` output should be removable without affecting tracked files.
- Nested packages are not part of this repo today, so a root ignore is sufficient.
- If future tools create other coverage folders, they should be handled in a separate cleanup.
