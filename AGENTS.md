## Project Guidance

This repo is a Bun-managed Expo template with reusable UI and media packages.
Do normal coding work directly unless the user explicitly asks for a plan.

### How To Work

- Read this file first, then load only the docs and source files needed for the task.
- Validate with fresh command output. Never claim tests, builds, UI checks, or CI passed from memory.
- Keep durable template guidance in `docs/` or the relevant package README.
- Keep reusable UI in `packages/ui`; keep reusable media contracts and processing in `packages/media`.
- App integrations stay under `client/`, `server/`, `app/api/`, or `shared/`.

### Tech Stack

| Layer | Tech | Notes |
|-------|------|-------|
| Package manager | Bun | `bun.lock`; CI installs with `bun install --frozen-lockfile` |

| Router | Expo Router 57 | Typed routes, API routes, server output, server middleware and data loaders enabled |
| Language | TypeScript 6 strict | Path alias `@/*` points at repo root |
| UI | `@mrmeg/expo-ui` workspace package | RN primitives, design tokens, theme state, reusable components |
| Media | `@mrmeg/expo-media` workspace package | Client hooks, processing helpers, S3/R2 server handlers |
| State/data | Zustand 5, TanStack React Query 5 | Persisted client stores; query defaults live in app providers |

| Observability | Sentry React Native | Runtime DSN and native upload config are separately env-gated |
| Testing | Jest 29, jest-expo, RNTL 13 | `bun run typecheck`, `lint`, `check:features`, `test:ci`, package gates |

### Project Notes

- Use the exact package scripts in `package.json`; do not substitute generic Expo or npm commands when a local Bun script exists.
- Quote route paths with parentheses or brackets in shell commands, for example `'app/(main)/(tabs)/index.tsx'`.
- Optional auth, billing, media, and Sentry features must fail closed when env is missing; a blank `.env` should keep the template explorable.
- For web SSR or hydration work, read `docs/ssr-hydration.md` before editing and verify with real server HTML, not only Jest or `tsc`.
- `llms-full.txt` and `llms-examples.txt` are generated by `scripts/build-llms-full.mjs` from its source-doc list and directory walks. After editing those docs or adding/removing demo routes, screens, components, or form wrappers, run `bun run docs:llms` and commit the result; CI runs `bun run docs:llms:check`.

## Agent Workflow

<!-- agent-framework:v2 -->

Planned work lives locally in the Git-ignored `Agent/` folder as one Markdown spec per task, with status and `base-branch` in frontmatter. The folder is the task list; implementation PRs never modify it.

- Specs are executed in fresh sessions, so each spec must stand alone. Read the spec and the source files it points to before starting planned work; never assume chat history.
- Use the `write-spec` skill to draft a spec and `review-spec` to promote it to ready or blocked. If `Agent/` is missing, run `bootstrap-docs` — the folder is disposable and safe to delete.
- The `shift` skill implements ready specs: one branch and one draft PR per spec, branched from and targeted at the spec's `base-branch`. `dayshift` and `nightshift` run it as a long session in a fixed mode; `review-shift` walks the open `agent/*` PRs.
- Keep the integration branch clean and synchronized before a shift starts and before an agent PR merges. Continue unrelated work on another branch or worktree, or commit and push it before review.
- Run `update-docs` periodically to prune stale specs and fix drifted notes.
- Validate with fresh command output before claiming tests, builds, or checks passed.
- For native mobile UI flows, prefer simulator/emulator verification with available tooling such as Argent; if the tooling or device is unavailable, record the blocker instead of treating the flow as verified.
- Ask only about irreversible decisions, genuine ambiguity, or blockers.
